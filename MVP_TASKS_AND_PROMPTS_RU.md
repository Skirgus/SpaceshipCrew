# Demo MVP: задачи и промты (RU)

## 1) Правила выполнения

- Базовая ветка: `feature/demo-mvp-implementation`.
- Каждая задача выполняется в отдельной ветке от базовой:
  - шаблон: `task/<ID>-<short-name>`.
- По окончании каждой задачи результат должен быть `PIE-ready`:
  - проект запускается в PIE без blocker/crash;
  - новый функционал задачи виден и проверяем;
  - базовый smoke-test пройден.
- На каждом шаге добавляем тесты:
  - минимум: 1-2 автоматических теста логики (где возможно) + чеклист ручной проверки PIE.
- После фактического закрытия задачи **обязательно обнови этот файл**:
  - в разделе **3)** у соответствующей задачи добавь строку `- **Статус:** выполнено` (при желании кратко: ветка/PR/дата);
  - в разделе **6)** у соответствующего промта в заголовке пометь **выполнено** (например: `### Prompt T01 (v2) — выполнено`), чтобы статус backlog и промтов не расходился.

## 2) Ограничения по контенту (без художников)

- Если нет готовых ассетов, используем:
  1. Примитивы Unreal (`Cube`, `Sphere`, `Cylinder`, `Plane`),
  2. Geometry Script / BSP / Modeling Tools,
  3. Бесплатные ассеты (Fab/Marketplace/Quixel) с совместимой лицензией.
- Для MVP допустим greybox-уровень визуала.
- Для каждого внешнего ассета фиксировать источник и лицензию в `Content/THIRD_PARTY_ASSETS.md`.

## 3) Декомпозиция задач (конкретный backlog)

## T01 - Главное меню
- **Статус:** выполнено
- Ветка: `task/T01-main-menu`
- Зависимости: нет
- Результат: экран с кнопками `Конструктор`, `Тренировки`, `Новая игра`, `Настройки`, `Выход`.
- PIE критерий: открытие проекта -> меню загрузилось -> все кнопки активны.
- Тесты:
  - автотест на наличие маршрутов/экранов меню (если реализовано через UI router/data table),

## T02a - Пайплайн создания модулей корабля (editor tooling)
- Ветка: `task/T02a-module-authoring-pipeline`
- Зависимости: T01
- Результат: быстрый и удобный поток создания новых модулей для конструктора через данные и шаблоны.
- PIE критерий: новый модуль можно создать по шаблону (editor tool), провалидировать; запись попадает в **единый каталог модулей** (источник данных для будущего конструктора); в PIE модуль **читается из каталога** (допускается минимальный дебаг-экран/список или существующий тестовый entry — без UI полноценного конструктора и без размещения в сборке — это **T02c**, логика — **T02b**).
- Тесты:
  - автотест обязательных полей `ModuleDefinition`,
  - автотест валидации сокетов/контактных точек.

## T02b - Домен сборки корабля (логика без сцены конструктора)
- Ветка: `task/T02b-ship-build-domain`
- Зависимости: T02a
- Результат: runtime-модель текущей сборки (список установленных модулей, привязки к сокетам/якорям), чтение **каталога модулей T02a**, **валидация** совместимости контактных точек, **расчёт суммарной массы**; публичный API «добавить / удалить / заменить модуль в конфигурации».
- Non-goals: без полноценного UI конструктора и без обязательной 3D-сцены сборки (это T02c); без сохранения пресета (T03).
- PIE критерий: через минимальный тестовый слой (команда, дебаг-виджет, автотестовый world) можно собрать **валидную** и **невалидную** конфигурацию и получить корректные флаги валидации и массу; модуль, созданный пайплайном T02a, **участвует в расчётах** (те же данные, что увидит конструктор).
- Тесты:
  - автотест валидации контактных точек,
  - автотест расчёта массы.

## T02c - Сцена и UI конструктора корабля (родительская задача)
- Ветка (суммарно): `task/T02c-ship-builder-scene-ui`; фактическая реализация ведётся **подзадачами T02c-1 → T02c-2 → T02c-3** (ниже).
- Зависимости: T01, T02b (плюс цепочка подзадач).
- Результат: сцена/подуровень конструктора, вход из главного меню, **UI каталога** (читает каталог T02a через слой T02b), визуальное размещение и снятие модулей, вызов API T02b для валидации и итоговой массы; после T02c-2 — **превью в UI и меш в конструкторе** из данных каталога; задел под **интерьер и перемещение внутри собранного корабля** (без полноценной навигации в рамках T02c).
- PIE критерий (закрывается по завершении **всех** подзадач): можно собрать валидный корабль из базового набора; **модуль из T02a виден в UI каталога и ставится в сборку** (интеграция «инструмент → каталог → сборка»); превью и меш конструктора задаются инструментом T02c-2 и используются в сцене.
- Тесты (по подзадачам — см. промты T02c-1…T02c-3):
  - ручной PIE smoke по сценарию конструктора;
  - по возможности узкий автотест на связку UI→T02b или загрузку presentation-полей каталога.

### T02c-1 - Уровень конструктора, UI, черновой конструктор (плейсхолдеры данных)
- **Статус:** ✅ **выполнено** (2026-04-16). PR: https://github.com/Skirgus/SpaceshipCrew/pull/16
- Ветка: `task/T02c1-ship-builder-level-ui`
- Зависимости: T01, T02b
- **Фокус задачи — каркас UI и расширяемая модель параметров корабля** (нижняя панель, блок реактора/энергии по референсу, чеклист, каталог по типам модуля). Конкретные числа, формулы баланса и полный набор правил «что улучшается/ухудшается» **не фиксируются** в этой задаче: нужны **точки расширения** — реестр/провайдер параметров (идентификаторы, например `FName`), **снимки** значений, сравнение двух снимков для **предпросмотра дельт** при наведении на строку каталога; UI строит полосы **итерацией по списку параметров**, а не отдельным виджетом на каждый стат.
- **Непосредственная стыковка модулей в мире** (выбор сокетов, граф `FShipBuildModuleConnection`, игровой UX с `AddAttachedModule` / размещение по стыкам) — **вне scope T02c-1**, выносится в **отдельную задачу**. Для экрана достаточно **чернового состояния конфигурации** (например список учтённых `ModuleId` или упрощённая модель без графа соединений), согласованного с будущей стыковкой.
- Результат: уровень конструктора, **`OpenLevel` из главного меню** (T01), базовые классы **UMG** (наследники `USpaceshipCrewMenuWidgetBase`) с начальной разметкой в коде в духе текущего меню; **каталог** из `UShipModuleCatalog`, группировка по **`EShipModuleType`**; виджет **чеклиста** (отдельно **ошибки** и **предупреждения**); инфо о модуле и предпросмотр дельт параметров при hover; по желанию клик по модулю фиксирует его в черновике **без** стыковки в сцене.
- **T02b (минимальная поверхность под UI):** расширить `FShipBuildValidationResult`: массив **`Warnings`** наряду с **`Errors`**; `bIsValid` от **ошибок** (предупреждения формально не блокируют); тексты могут быть **заглушечными** — важно стабильное **разделение каналов** для виджета. Масса и существующая валидация T02b остаются источниками правды там, где уже реализованы.
- **Вьюпорт:** опционально один **визуальный** плейсхолдер или «призрак» (единый меш) как якорь; **не** требуется DoD на полноценное размещение/снятие через API T02b в PIE. Уникальные превью/меши на модуль — **T02c-2 / T02c-3**.
- Non-goals: сохранение пресета (T03); NavMesh и ходьба внутри корабля; **полная стыковка модулей** (отдельная задача); фиксация финального баланса параметров.
- PIE критерий: меню → конструктор → назад; каталог не пуст при наличии модулей; чеклист различает ошибки и предупреждения; параметры отображаются через **расширяемую** схему (новый параметр = запись в реестре/провайдере, без копипасты виджетов); подключена валидация T02b в части **Errors/Warnings** и массы где применимо.
- Тесты: ручной PIE smoke; по возможности автотест на пустую сборку / разделение **Error** vs **Warning**; иначе обоснование в отчёте.

### T02c-2 - Данные и инструмент: превью, меш конструктора, задел под интерьер корабля
- Ветка: `task/T02c2-ship-builder-module-presentation`
- Зависимости: T02a, T02c-1
- Результат: расширение **ModuleDefinition** / каталога (мягкие ссылки): **текстура превью для UI конструктора**, **StaticMesh для сцены конструктора** (призрак и размещённый модуль). Обновление **editor-пайплайна** (на базе T02a): назначение превью и меша, валидация обязательности для «готового» модуля. **Контракт под будущее перемещение внутри корабля:** единый pivot и сокеты/контакты согласованы с T02b; `hasInterior`; опциональные поля под интерьер (soft ref или идентификатор) — без реализации NavMesh и геймплея ходьбы в этой задаче. Краткий **документ с конвенцией** (в репозитории рядом с данными или в комментариях к типу). Рантайм: каталог отдаёт превью и меш конструктору без дублирования путей в виджетах.
- PIE критерий: модуль можно «оформить» в инструменте (картинка + меш); в PIE каталог показывает превью; в сцене используется назначенный меш (замена унифицированного плейсхолдера — полная интеграция по всем слотам может завершаться в T02c-3).
- Тесты: автотест на наличие/загрузку presentation-полей или сериализацию; ручной прогон editor → PIE.

### T02c-3 - Режим строительства: каталожные меши вместо плейсхолдеров, полировка UX

- PIE критерий: модуль, созданный пайплайном T02a и оформленный в T02c-2, **виден в каталоге с превью и в сборке с корректным мешем**; сценарии валид/невалид проходятся вручную.

Использовать этот шаблон для любой задачи T01-T12 и подзадач конструктора **T02a / T02b / T02c / T02c-1 / T02c-2 / T02c-3**.

```text
Задача: <ID и название>

ЭТАП 1 - PLAN (обязательно)
1) Кратко опиши план реализации (5-10 пунктов).

ЭТАП 2 - IMPLEMENTATION (обязательно)
1) Реализуй только scope этой задачи.
2) Не выходи за границы Non-goals.
3) Добавь минимум 1-2 автотеста, где применимо.

1) Запусти автопроверки/тесты для измененной логики.

   - Что сделано
   - Что проверено (авто + ручные тесты)

   - Следующий шаг
```

## 5) Единые QA-правила и формат отчета

- Любая задача считается выполненной только при `PIE-ready`.
- После закрытия задачи актуализируй маркеры **выполнено** в разделах **3)** и **6)** этого документа (см. правила в п. 1).
- Минимум тестов на задачу:
  - 1 автотест логики (или обоснование, почему невозможно),
  - 1 ручной PIE smoke-check.
- Любые использованные внешние ассеты фиксировать в `Content/THIRD_PARTY_ASSETS.md`.
- Запрещено расширять scope без явной фиксации в задаче.
- Формат финального отчета обязателен:
  - `Что сделано`
  - `Что проверено`
  - `Что не закрыто / риски`
  - `Следующий шаг`
