# Demo MVP: задачи и промты (RU)

## 1) Правила выполнения

- Базовая ветка: `feature/demo-mvp-implementation`.
- Каждая задача выполняется в отдельной ветке от базовой:
  - шаблон: `task/<ID>-<short-name>`.
- По окончании каждой задачи результат должен быть `PIE-ready`:
  - проект запускается в PIE без blocker/crash;
  - новый функционал задачи виден и проверяем;
  - базовый smoke-test пройден.
- На каждом шаге добавляем тесты:
  - минимум: 1-2 автоматических теста логики (где возможно) + чеклист ручной проверки PIE.
- После фактического закрытия задачи **обязательно обнови этот файл**:
  - в разделе **3)** у соответствующей задачи добавь строку `- **Статус:** выполнено` (при желании кратко: ветка/PR/дата);
  - в разделе **6)** у соответствующего промта в заголовке пометь **выполнено** (например: `### Prompt T01 (v2) — выполнено`), чтобы статус backlog и промтов не расходился.

## 2) Ограничения по контенту (без художников)

- Если нет готовых ассетов, используем:
  1. Примитивы Unreal (`Cube`, `Sphere`, `Cylinder`, `Plane`),
  2. Geometry Script / BSP / Modeling Tools,
  3. Бесплатные ассеты (Fab/Marketplace/Quixel) с совместимой лицензией.
- Для MVP допустим greybox-уровень визуала.
- Для каждого внешнего ассета фиксировать источник и лицензию в `Content/THIRD_PARTY_ASSETS.md`.

## 3) Декомпозиция задач (конкретный backlog)

## T01 - Главное меню
- **Статус:** выполнено
- Ветка: `task/T01-main-menu`
- Зависимости: нет
- Результат: экран с кнопками `Конструктор`, `Тренировки`, `Новая игра`, `Настройки`, `Выход`.
- PIE критерий: открытие проекта -> меню загрузилось -> все кнопки активны.
- Тесты:
  - автотест на наличие маршрутов/экранов меню (если реализовано через UI router/data table),
  - ручной smoke: переход по каждой кнопке.

## T02a - Пайплайн создания модулей корабля (editor tooling)
- Ветка: `task/T02a-module-authoring-pipeline`
- Зависимости: T01
- Результат: быстрый и удобный поток создания новых модулей для конструктора через данные и шаблоны.
- PIE критерий: новый модуль можно создать по шаблону (editor tool), провалидировать; запись попадает в **единый каталог модулей** (источник данных для будущего конструктора); в PIE модуль **читается из каталога** (допускается минимальный дебаг-экран/список или существующий тестовый entry — без UI полноценного конструктора и без размещения в сборке — это **T02c**, логика — **T02b**).
- Тесты:
  - автотест обязательных полей `ModuleDefinition`,
  - автотест валидации сокетов/контактных точек.

## T02b - Домен сборки корабля (логика без сцены конструктора)
- Ветка: `task/T02b-ship-build-domain`
- Зависимости: T02a
- Результат: runtime-модель текущей сборки (список установленных модулей, привязки к сокетам/якорям), чтение **каталога модулей T02a**, **валидация** совместимости контактных точек, **расчёт суммарной массы**; публичный API «добавить / удалить / заменить модуль в конфигурации».
- Non-goals: без полноценного UI конструктора и без обязательной 3D-сцены сборки (это T02c); без сохранения пресета (T03).
- PIE критерий: через минимальный тестовый слой (команда, дебаг-виджет, автотестовый world) можно собрать **валидную** и **невалидную** конфигурацию и получить корректные флаги валидации и массу; модуль, созданный пайплайном T02a, **участвует в расчётах** (те же данные, что увидит конструктор).
- Тесты:
  - автотест валидации контактных точек,
  - автотест расчёта массы.

## T02c - Сцена и UI конструктора корабля
- Ветка: `task/T02c-ship-builder-scene-ui`
- Зависимости: T01, T02b
- Результат: сцена/подуровень конструктора, вход из главного меню, **UI каталога** (читает каталог T02a через слой T02b), визуальное размещение и снятие модулей (акторы/примитивы), вызов API T02b для валидации и итоговой массы.
- PIE критерий: можно собрать валидный корабль из базового набора; **модуль из T02a виден в UI каталога и ставится в сборку** (интеграция «инструмент → каталог → сборка»).
- Тесты:
  - ручной PIE smoke по сценарию конструктора;
  - по возможности один узкий автотест на связку «команда UI → вызов T02b» (если нет — обоснование в отчёте).

## T03 - Сохранение/загрузка пресета корабля
- Ветка: `task/T03-ship-preset-save`
- Зависимости: T02c
- Результат: сохранение кастомного корабля и отображение в списке пресетов.
- PIE критерий: собрать -> сохранить -> закрыть/открыть -> пресет доступен.
- Тесты:
  - автотест сериализации/десериализации пресета,
  - автотест валидации бюджета на загруженном пресете.

## T04 - Старт новой кампании (корабль + бюджет + роль)
- Ветка: `task/T04-new-game-flow`
- Зависимости: T01, T03
- Результат: запуск новой игры с выбором пресета/кастома, проверкой бюджета, выбором роли.
- PIE критерий: валидный выбор запускает игру; невалидный блокируется.
- Тесты:
  - автотест budget gate,
  - ручной smoke: роль по умолчанию капитан, смена роли работает.

## T05 - Стартовая станция и базовые интеракции
- Ветка: `task/T05-start-station`
- Зависимости: T04
- Результат: загрузка станции, точки взаимодействия (терминалы/NPC-заглушки).
- PIE критерий: первый вход на станцию стабильный.
- Тесты:
  - автотест загрузки уровня и ключевых акторов,
  - ручной smoke взаимодействий.

## T06 - UI станций v1 (капитан/пилот/стрелок/инженер)
- Ветка: `task/T06-station-ui-v1`
- Зависимости: T05

- PIE критерий: каждая станция выполняет минимум 1 ключевое действие.
- Тесты:
  - автотесты данных UI-моделей (если MVVM),
  - ручной сценарий mini UI flow.

## T07 - Тренировки по ролям
- Ветка: `task/T07-role-trainings`
- Зависимости: T06
- Результат: 4 тренировочные миссии, доступные в любом порядке.
- PIE критерий: каждая тренировка завершается по условию.
- Тесты:
  - автотест условий завершения тренировки,
  - ручной прогон всех 4 тренировок.

## T08 - Найм и назначение ботов
- Ветка: `task/T08-bot-hiring`
- Зависимости: T05, T06
- Результат: список ботов, найм, назначение на роли/станции.
- PIE критерий: старт вылета с неполной командой закрывается ботами.
- Тесты:
  - автотест расчета стоимости найма,
  - автотест назначения роли боту.

## T09 - Экономический расчет рейса
- Ветка: `task/T09-run-economy`
- Зависимости: T08

- Тесты:
  - автотест формулы `RealizedIncome/ShipFund/CrewSalaryPayout`,

- Ветка: `task/T10-autosave-rejoin`
- Зависимости: T09
- Результат: autosave перед вылетом, rejoin со снапшотом состояния.
- PIE критерий: переподключение не дублирует ресурсы и восстанавливает инвентарь.

## T11 - Vote-kick и анти-абьюз baseline
- Ветка: `task/T11-votekick-antabuse`

- PIE критерий: vote-kick работает по порогу голосов и не ломает сессию.
- Тесты:
  - автотест cooldown/threshold логики,
  - ручной кооп smoke (2-4 клиента).

## T12 - Demo polish + first-pass баланс
- Ветка: `task/T12-demo-polish-balance`
- Зависимости: T11
- Результат: стабильный demo-путь + первичная настройка коэффициентов по Playtest Sheet.
- PIE критерий: 3 последовательных прогона demo без blocker/crash.
- Тесты:
  - regression smoke checklist,
  - фиксация KPI рейсов (минимум 12 прогонов).

## 4) Строгий шаблон промта v2 (3 этапа)

Использовать этот шаблон для любой задачи T01-T12 и подзадач конструктора **T02a / T02b / T02c**.

```text
Задача: <ID и название>
Ветка: <task/...> от feature/demo-mvp-implementation

ЭТАП 1 - PLAN (обязательно)
1) Кратко опиши план реализации (5-10 пунктов).
2) Перечисли затрагиваемые подсистемы/файлы.
3) Укажи риски и как их обойти.
4) Зафиксируй Non-goals (что не делаем в этой задаче).
5) Подтверди критерии DoD и PIE-ready перед началом кодирования.

ЭТАП 2 - IMPLEMENTATION (обязательно)
1) Реализуй только scope этой задачи.
2) Не выходи за границы Non-goals.
3) Добавь минимум 1-2 автотеста, где применимо.
4) Убедись, что проект запускается в PIE и новая функциональность видна.
5) Если встретился блокер, опиши rollback/mitigation шаги и минимально безопасный скоуп.

ЭТАП 3 - REVIEW/QA (обязательно)
1) Запусти автопроверки/тесты для измененной логики.
2) Прогони ручной PIE smoke-check по чеклисту задачи.
3) Дай отчет в формате:
   - Что сделано
   - Что проверено (авто + ручные тесты)
   - Что не закрыто / риски
   - Следующий шаг
```

## 5) Единые QA-правила и формат отчета

- Любая задача считается выполненной только при `PIE-ready`.
- После закрытия задачи актуализируй маркеры **выполнено** в разделах **3)** и **6)** этого документа (см. правила в п. 1).
- Минимум тестов на задачу:
  - 1 автотест логики (или обоснование, почему невозможно),
  - 1 ручной PIE smoke-check.
- Любые использованные внешние ассеты фиксировать в `Content/THIRD_PARTY_ASSETS.md`.
- Запрещено расширять scope без явной фиксации в задаче.
- Формат финального отчета обязателен:
  - `Что сделано`
  - `Что проверено`
  - `Что не закрыто / риски`
  - `Следующий шаг`
